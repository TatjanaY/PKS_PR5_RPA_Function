# Вариант 16

## Кейс: Автоматизация формирования списка приоритетных клиентов

В крупной компании внедряется система автоматического выделения приоритетных клиентов для персонального обслуживания.
Руководство приняло решение исключить ручную сегментацию клиентской базы и минимизировать риски упущенных возможностей при работе с наиболее ценными партнёрами.

Ранее менеджеры вручную анализировали объёмы покупок, частоту заказов и длительность сотрудничества, формируя списки для персональных предложений. Процесс занимал значительное время и сопровождался риском субъективной оценки и пропуска ключевых клиентов.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел продаж формирует список клиентов с данными о взаимодействиях и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список клиентов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "client_name": "ООО «Вектор»",
  "total_spent": 750000,
  "orders_per_year": 24,
  "cooperation_months": 18
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка минимального порога расходов

### Ситуация

На первом этапе система должна проверить, превышает ли клиент минимальный порог суммарных расходов для рассмотрения в качестве приоритетного.
Порог установлен на уровне 500 000 рублей.

Отделу продаж необходимо получить отдельный файл со статусом соответствия порогу по каждому клиенту.

---

## Требуется:

1. Считать JSON-файл (6 клиентов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckSpendingThreshold`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `client_dict`

**Логика:**

* Получить значение `total_spent` из словаря.
* Если `total_spent` < 500000 → вернуть `"Не соответствует"`.
* Если `total_spent` ≥ 500000 → вернуть `"Соответствует"`.

---

## Главный процесс должен:

1. Создать пустой список `threshold_result_list`.
2. В цикле:

   * Вызвать `CheckSpendingThreshold`;
   * Сформировать словарь:

```json
{
  "client_name": "ООО «Вектор»",
  "threshold_status": "Соответствует"
}
```

* Добавить словарь в `threshold_result_list`.

3. Сохранить `threshold_result_list` в файл
   `threshold_result.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку порога расходов.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт приоритетного балла и формирование итогового статуса

### Ситуация

После проверки порога расходов руководство решило автоматизировать комплексную оценку ценности клиента.

Теперь требуется не только подтвердить соответствие порогу, но и рассчитать интегральный приоритетный балл на основе трёх факторов (расходы, частота заказов, длительность сотрудничества), а также присвоить итоговый статус приоритетности.

Отделу развития клиентов нужен структурированный список с расчётными данными для планирования персональных коммуникаций.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 клиентов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculatePriorityScore`

**Параметр:**

* `client_dict`

**Логика:**

* Получить `total_spent`, `orders_per_year`, `cooperation_months`.
* Рассчитать балл по формуле:
  `(total_spent / 100000) + (orders_per_year * 0.5) + (cooperation_months / 6)`
* Вернуть число (приоритетный балл).

---

### 2️⃣ Функция `DeterminePriorityStatus`

**Параметр:**

* `priority_score`

**Логика:**

* Если `priority_score` ≥ 20 → вернуть `"Высокий приоритет"`
* Если `priority_score` ≥ 12 → вернуть `"Средний приоритет"`
* Иначе → вернуть `"Низкий приоритет"`

---

### 3️⃣ Функция `BuildPriorityRecord`

**Параметры:**

* `client_name`
* `priority_status`
* `score_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "client_name": "ООО «Вектор»",
  "priority_status": "Высокий приоритет",
  "priority_score": 23.5
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_priority_list`.
2. В цикле для каждого клиента:

   * Вызвать `CalculatePriorityScore`;
   * Передать результат в `DeterminePriorityStatus`;
   * Вызвать `BuildPriorityRecord`;
   * Добавить возвращённый словарь в `final_priority_list`.
3. Сохранить `final_priority_list` в файл
   `priority_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
