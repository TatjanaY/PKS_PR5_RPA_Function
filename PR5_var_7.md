# Вариант 7

## Кейс: Автоматизация проверки корректности заявок на оплату

В крупной компании внедряется система автоматической валидации заявок на оплату счетов и расходов.
Руководство приняло решение исключить ручную проверку финансовых документов и минимизировать риски оплаты некорректных или подозрительных заявок.

Ранее сотрудники финансового отдела вручную сверяли суммы заявок с лимитами, проверяли наличие необходимых реквизитов и соответствие статьям бюджета. Процесс занимал значительное время и сопровождался риском пропуска ошибок в реквизитах или превышения лимитов.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Финансовый отдел формирует список заявок на оплату и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список заявок;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "request_id": "PAY-001",
  "amount": 15000.00,
  "has_all_requisites": true,
  "budget_article": "Office"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка наличия реквизитов

### Ситуация

На первом этапе система должна проверить, заполнены ли все обязательные реквизиты в заявке на оплату.
Заявки с отсутствующими реквизитами не могут быть обработаны и требуют доработки.

Финансовому отделу необходимо получить отдельный файл со статусом комплектности документов по каждой заявке.

---

## Требуется:

1. Считать JSON-файл (6 заявок).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckRequisitesCompleteness`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `request_dict`

**Логика:**

* Получить значение `has_all_requisites` из словаря.
* Если `has_all_requisites` = `false` → вернуть `"Требует доработки"`.
* Если `has_all_requisites` = `true` → вернуть `"Комплектна"`.

---

## Главный процесс должен:

1. Создать пустой список `requisites_check_list`.
2. В цикле:

   * Вызвать `CheckRequisitesCompleteness`;
   * Сформировать словарь:

```json
{
  "request_id": "PAY-001",
  "requisites_status": "Комплектна"
}
```

* Добавить словарь в `requisites_check_list`.

3. Сохранить `requisites_check_list` в файл
   `requisites_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку комплектности реквизитов.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Проверка суммы и формирование итогового статуса оплаты

### Ситуация

После проверки реквизитов руководство решило усложнить процесс валидации.

Теперь требуется также проверить сумму заявки на соответствие лимитам и сформировать итоговый статус для передачи в платёжную систему.

Финансовому отделу нужен структурированный список заявок с финальными статусами для автоматической обработки платежей.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 заявок.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CheckAmountLimit`

**Параметр:**

* `request_dict`

**Логика:**

* Получить значение `amount`.
* Если `amount` > 100000 → вернуть `false` (превышен лимит).
* Если `amount` ≤ 100000 → вернуть `true` (в пределах лимита).

---

### 2️⃣ Функция `DeterminePaymentStatus`

**Параметры:**

* `requisites_ok` (булево)
* `amount_ok` (булево)

**Логика:**

* Если `requisites_ok` = `true` И `amount_ok` = `true` → вернуть `"Одобрено"`
* Если `requisites_ok` = `false` → вернуть `"На доработку"`
* Иначе → вернуть `"На согласование"`

---

### 3️⃣ Функция `BuildPaymentRecord`

**Параметры:**

* `request_id`
* `payment_status`
* `amount_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "request_id": "PAY-001",
  "payment_status": "Одобрено",
  "amount": 15000.00
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_payment_list`.
2. В цикле для каждой заявки:

   * Проверить `has_all_requisites` (из исходных данных);
   * Вызвать `CheckAmountLimit`;
   * Вызвать `DeterminePaymentStatus` с двумя булевыми параметрами;
   * Вызвать `BuildPaymentRecord`;
   * Добавить возвращённый словарь в `final_payment_list`.
3. Сохранить `final_payment_list` в файл
   `payment_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
