# Вариант 22

## Кейс: Автоматизация анализа задержек поставок

В крупной логистической компании внедряется система автоматического мониторинга своевременности поставок от контрагентов.
Руководство приняло решение исключить ручную сверку плановых и фактических дат доставки и минимизировать риски срывов производственных графиков из-за несвоевременных поставок.

Ранее специалисты отдела снабжения вручную анализировали накладные, сравнивали даты и рассчитывали штрафы за просрочку. Процесс занимал значительное время при закрытии отчётного периода и сопровождался риском ошибок при работе с большими объёмами данных.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел логистики формирует список поставок с данными о сроках и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список поставок;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "supplier_name": "ООО «Логистик-Плюс»",
  "planned_delivery_date": "2024-01-20",
  "actual_delivery_date": "2024-01-23",
  "order_value": 150000
}
```

5. Робот должен считать данные из созданного JSON-файла.

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка своевременности доставки

### Ситуация

На первом этапе система должна проверить, была ли поставка осуществлена в срок.
Поставка считается своевременной, если фактическая дата доставки не позже плановой.

Отделу снабжения необходимо получить отдельный файл со статусом своевременности по каждой поставке.

---

## Требуется:

1. Считать JSON-файл (6 поставок).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckDeliveryTimeliness`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `delivery_dict`

**Логика:**

* Получить значения `planned_delivery_date` и `actual_delivery_date`.
* Для упрощения: сравнить строки дат лексикографически (формат ГГГГ-ММ-ДД).
* Если `actual_delivery_date` > `planned_delivery_date` → вернуть `"Задержка"`.
* Иначе → вернуть `"В срок"`.

---

## Главный процесс должен:

1. Создать пустой список `timeliness_check_list`.
2. В цикле:

   * Вызвать `CheckDeliveryTimeliness`;
   * Сформировать словарь:

```json
{
  "supplier_name": "ООО «Логистик-Плюс»",
  "delivery_status": "В срок"
}
```

* Добавить словарь в `timeliness_check_list`.

3. Сохранить `timeliness_check_list` в файл
   `timeliness_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку своевременности.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт штрафа и формирование итогового статуса

### Ситуация

После проверки своевременности руководство решило автоматизировать расчёт финансовых последствий задержек.

Теперь требуется не только зафиксировать факт задержки, но и рассчитать сумму штрафа (1% от стоимости заказа за каждый день просрочки), а также присвоить итоговый статус для проведения взаиморасчётов.

Финансовому отделу нужен структурированный список поставок с расчётными данными для корректировки платежей контрагентам.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 поставок.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateDelayDays`

**Параметр:**

* `delivery_dict`

**Логика:**

* Получить `planned_delivery_date` и `actual_delivery_date`.
* Для упрощения: извлечь день из каждой даты и рассчитать разницу.
* Если разница ≤ 0 → вернуть 0.
* Иначе → вернуть количество дней задержки.

---

### 2️⃣ Функция `CalculatePenaltyAmount`

**Параметры:**

* `order_value`
* `delay_days`

**Логика:**

* Рассчитать штраф:
  `order_value * 0.01 * delay_days`
* Вернуть число (сумма штрафа).

---

### 3️⃣ Функция `BuildDeliveryRecord`

**Параметры:**

* `supplier_name`
* `delivery_status`
* `penalty_amount`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "supplier_name": "ООО «Логистик-Плюс»",
  "delivery_status": "Задержка",
  "penalty_amount": 4500
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_delivery_list`.
2. В цикле для каждой поставки:

   * Вызвать `CheckDeliveryTimeliness` (для статуса);
   * Вызвать `CalculateDelayDays`;
   * Передать результат и `order_value` в `CalculatePenaltyAmount`;
   * Вызвать `BuildDeliveryRecord`;
   * Добавить возвращённый словарь в `final_delivery_list`.
3. Сохранить `final_delivery_list` в файл
   `delivery_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
