# Вариант 35

## Кейс: Автоматизация проверки соответствия версий программного обеспечения

В крупной компании внедряется система автоматического контроля версий программного обеспечения на рабочих станциях сотрудников.
Руководство приняло решение исключить ручную сверку установленных версий с корпоративными стандартами и минимизировать риски использования устаревшего или нелицензионного ПО.

Ранее сотрудники ИТ-отдела вручную собирали данные о версиях программ, сравнивали их с утверждённым реестром и формировали списки на обновление. Процесс занимал значительное время при подготовке аудитов и сопровождался риском пропуска критических обновлений безопасности.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

ИТ-отдел формирует список рабочих станций с данными об установленном ПО и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список рабочих станций;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "workstation_id": "WS-001",
  "software_name": "Office Suite",
  "installed_version": "2021.3",
  "required_version": "2023.1",
  "license_status": "active"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка соответствия версии ПО

### Ситуация

На первом этапе система должна проверить, соответствует ли установленная версия программы требуемой корпоративной версии.
Рабочие станции с устаревшими версиями ПО требуют обновления для обеспечения безопасности и совместимости.

ИТ-отделу необходимо получить отдельный файл со статусом соответствия версий по каждой рабочей станции.

---

## Требуется:

1. Считать JSON-файл (6 записей).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckVersionCompliance`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `software_dict`

**Логика:**

* Получить значения `installed_version` и `required_version` из словаря.
* Сравнить версии (для упрощения: сравнение строк или числовых суффиксов).
* Если `installed_version` < `required_version` → вернуть `"Требуется обновление"`.
* Если `installed_version` ≥ `required_version` → вернуть `"Версия актуальна"`.

---

## Главный процесс должен:

1. Создать пустой список `version_check_list`.
2. В цикле:

   * Вызвать `CheckVersionCompliance`;
   * Сформировать словарь:

```json
{
  "workstation_id": "WS-001",
  "version_status": "Требуется обновление"
}
```

* Добавить словарь в `version_check_list`.

3. Сохранить `version_check_list` в файл
   `version_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку соответствия версии.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт приоритета обновления и формирование задачи на установку

### Ситуация

После проверки версий руководство решило усложнить процесс управления обновлениями.

Теперь требуется рассчитать приоритет обновления на основе разницы версий, статуса лицензии и критичности ПО, а также сформировать задачу для системы автоматического развёртывания.

ИТ-отделу нужен структурированный список рабочих станций с приоритетами для планирования обновлений в нерабочее время.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 записей.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateUpdatePriority`

**Параметр:**

* `software_dict`

**Логика:**

* Получить `installed_version`, `required_version`, `license_status`.
* Базовый приоритет = 10.
* Если `license_status` = "expired" → добавить 40 баллов.
* Если разница версий ≥ 2 (условно) → добавить 30 баллов.
* Если разница версий = 1 → добавить 15 баллов.
* Вернуть итоговый приоритет.

---

### 2️⃣ Функция `DetermineUpdateAction`

**Параметр:**

* `priority_value`

**Логика:**

* Если `priority_value` ≥ 50 → вернуть `"Срочное обновление"`.
* Если `priority_value` ≥ 30 → вернуть `"Плановое обновление"`.
* Если `priority_value` ≥ 10 → вернуть `"Отложить обновление"`.
* Иначе → вернуть `"Обновление не требуется"`.

---

### 3️⃣ Функция `BuildUpdateRecord`

**Параметры:**

* `workstation_id`
* `update_action`
* `priority_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "workstation_id": "WS-001",
  "update_action": "Срочное обновление",
  "priority_score": 65
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_update_list`.
2. В цикле для каждой записи:

   * Вызвать `CalculateUpdatePriority`;
   * Передать результат в `DetermineUpdateAction`;
   * Вызвать `BuildUpdateRecord`;
   * Добавить возвращённый словарь в `final_update_list`.
3. Сохранить `final_update_list` в файл
   `update_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
