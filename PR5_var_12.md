# Вариант 12

## Кейс: Автоматизация проверки корректности начисления бонусов

В крупной компании внедряется система автоматического контроля правильности начисления премиальных бонусов сотрудникам.
Руководство приняло решение исключить ручную сверку плановых и фактических показателей при выплате бонусов и минимизировать риски финансовых ошибок.

Ранее бухгалтеры вручную проверяли выполнение условий премирования, сверяли суммы с утверждёнными планами и корректировали выплаты. Процесс занимал значительное время в конце расчётного периода и сопровождался риском человеческих ошибок.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Финансовый отдел формирует список сотрудников с данными о бонусах и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список сотрудников;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "employee_name": "Орлов О.О.",
  "bonus_plan": 100000,
  "bonus_fact": 100000,
  "condition_met": true
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка соответствия условия премирования

### Ситуация

На первом этапе система должна проверить, было ли выполнено условие для начисления бонуса.
Бонус может быть начислен только если флаг `condition_met` установлен в `true`.

Финансовому отделу необходимо получить отдельный файл со статусом выполнения условия по каждому сотруднику.

---

## Требуется:

1. Считать JSON-файл (6 сотрудников).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckBonusCondition`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `employee_dict`

**Логика:**

* Получить значение `condition_met` из словаря.
* Если `condition_met` = `false` → вернуть `"Условие не выполнено"`.
* Если `condition_met` = `true` → вернуть `"Условие выполнено"`.

---

## Главный процесс должен:

1. Создать пустой список `condition_check_list`.
2. В цикле:

   * Вызвать `CheckBonusCondition`;
   * Сформировать словарь:

```json
{
  "employee_name": "Орлов О.О.",
  "condition_status": "Условие выполнено"
}
```

* Добавить словарь в `condition_check_list`.

3. Сохранить `condition_check_list` в файл
   `condition_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку флага условия.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Проверка корректности суммы и формирование статуса выплаты

### Ситуация

После проверки условия руководство решило автоматизировать контроль суммы начисления.

Теперь требуется не только подтвердить условие, но и сверить плановую сумму с фактической, а также присвоить статус корректности начисления для проведения платежа.

Отделу заработной платы нужен структурированный список сотрудников с финальными статусами для обработки выплат.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 сотрудников.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateBonusDiscrepancy`

**Параметр:**

* `employee_dict`

**Логика:**

* Получить `bonus_plan` и `bonus_fact`.
* Рассчитать отклонение:
  `bonus_fact - bonus_plan`
* Вернуть число (может быть отрицательным, нулевым или положительным).

---

### 2️⃣ Функция `DetermineAccrualStatus`

**Параметры:**

* `condition_met` (булево)
* `discrepancy_value` (число)

**Логика:**

* Если `condition_met` = `false` → вернуть `"Выплата запрещена"`.
* Если `condition_met` = `true` И `discrepancy_value` = 0 → вернуть `"Корректно"`.
* Если `condition_met` = `true` И `discrepancy_value` ≠ 0 → вернуть `"Требуется перерасчёт"`.

---

### 3️⃣ Функция `BuildAccrualRecord`

**Параметры:**

* `employee_name`
* `accrual_status`
* `discrepancy_amount`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "employee_name": "Орлов О.О.",
  "accrual_status": "Корректно",
  "discrepancy": 0
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_accrual_list`.
2. В цикле для каждого сотрудника:

   * Проверить `condition_met` (из исходных данных);
   * Вызвать `CalculateBonusDiscrepancy`;
   * Вызвать `DetermineAccrualStatus` с двумя параметрами;
   * Вызвать `BuildAccrualRecord`;
   * Добавить возвращённый словарь в `final_accrual_list`.
3. Сохранить `final_accrual_list` в файл
   `accrual_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
