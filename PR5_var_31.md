# Вариант 31

## Кейс: Автоматизация обработки заявок в Service Desk

В крупной IT-компании внедряется система автоматической обработки входящих заявок в службу технической поддержки (Service Desk).
Руководство приняло решение исключить ручную классификацию заявок и минимизировать риски ошибок при распределении обращений между специалистами.

Ранее сотрудники службы поддержки вручную анализировали описания проблем, определяли категории заявок и назначали исполнителей. Процесс занимал значительное время и сопровождался риском неверной классификации или назначения некомпетентного специалиста.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Служба поддержки формирует список заявок с данными о проблемах и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список заявок;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "ticket_id": "TKT-001",
  "category": "hardware",
  "priority": "high",
  "description": "Не работает принтер"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка категории заявки

### Ситуация

На первом этапе система должна проверить, указана ли корректная категория заявки.
Заявки без категории или с неизвестной категорией не могут быть автоматически распределены и требуют ручной обработки.

Службе поддержки необходимо получить отдельный файл со статусом категории по каждой заявке.

---

## Требуется:

1. Считать JSON-файл (6 заявок).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckTicketCategory`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `ticket_dict`

**Логика:**

* Получить значение `category` из словаря.
* Если `category` = "" или None → вернуть `"Категория не указана"`.
* Если `category` в списке допустимых (hardware, software, network, access) → вернуть `"Категория корректна"`.
* Иначе → вернуть `"Категория неизвестна"`.

---

## Главный процесс должен:

1. Создать пустой список `category_check_list`.
2. В цикле:

   * Вызвать `CheckTicketCategory`;
   * Сформировать словарь:

```json
{
  "ticket_id": "TKT-001",
  "category_status": "Категория корректна"
}
```

* Добавить словарь в `category_check_list`.

3. Сохранить `category_check_list` в файл
   `category_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку категории заявки.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт приоритета обработки и назначение исполнителя

### Ситуация

После проверки категории руководство решило усложнить процесс обработки заявок.

Теперь требуется рассчитать итоговый приоритет обработки на основе категории и заявленного приоритета, а также назначить соответствующую группу исполнителей для каждой заявки.

Службе поддержки нужен структурированный список заявок с назначенными исполнителями для автоматического распределения в систему учёта.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 заявок.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateProcessingPriority`

**Параметр:**

* `ticket_dict`

**Логика:**

* Получить `category` и `priority`.
* Базовый приоритет = 10.
* Если `priority` = "critical" → добавить 50 баллов.
* Если `priority` = "high" → добавить 30 баллов.
* Если `priority` = "medium" → добавить 20 баллов.
* Если `category` = "access" → добавить 10 баллов (доступы важны).
* Вернуть итоговый приоритет.

---

### 2️⃣ Функция `DetermineExecutorGroup`

**Параметр:**

* `category`

**Логика:**

* Если `category` = "hardware" → вернуть `"Группа инфраструктур"`.
* Если `category` = "software" → вернуть `"Группа разработки"`.
* Если `category` = "network" → вернуть `"Группа сетей"`.
* Если `category` = "access" → вернуть `"Группа безопасности"`.
* Иначе → вернуть `"Общая группа"`.

---

### 3️⃣ Функция `BuildTicketRecord`

**Параметры:**

* `ticket_id`
* `executor_group`
* `processing_priority`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "ticket_id": "TKT-001",
  "executor_group": "Группа инфраструктур",
  "processing_priority": 50
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_ticket_list`.
2. В цикле для каждой заявки:

   * Вызвать `CalculateProcessingPriority`;
   * Вызвать `DetermineExecutorGroup` (передать `category`);
   * Вызвать `BuildTicketRecord`;
   * Добавить возвращённый словарь в `final_ticket_list`.
3. Сохранить `final_ticket_list` в файл
   `ticket_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
