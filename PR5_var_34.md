# Вариант 34

## Кейс: Автоматизация формирования отчёта по исправленным ошибкам

В крупной IT-компании внедряется система автоматического формирования отчётов по исправленным ошибкам в программном продукте.
Руководство приняло решение исключить ручную агрегацию данных из системы отслеживания задач и минимизировать риски несогласованности отчётности перед релизами.

Ранее менеджеры проектов вручную собирали информацию о закрытых задачах, анализировали критичность ошибок и время их исправления, формируя отчёты в таблицах. Процесс занимал значительное время перед каждым релизом и сопровождался риском пропуска важных метрик качества.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел контроля качества формирует список исправленных ошибок с данными о характеристиках и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список ошибок;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "bug_id": "BUG-2024-001",
  "severity": "critical",
  "fix_time_hours": 8,
  "module_name": "authentication"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка критичности ошибки

### Ситуация

На первом этапе система должна проверить, относится ли исправленная ошибка к категории критических для включения в приоритетный отчёт.
Критические ошибки (severity = "critical") требуют обязательного отражения в отчёте для руководства.

Отделу качества необходимо получить отдельный файл со статусом критичности по каждой ошибке.

---

## Требуется:

1. Считать JSON-файл (6 ошибок).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckBugSeverity`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `bug_dict`

**Логика:**

* Получить значение `severity` из словаря.
* Если `severity` = "critical" → вернуть `"Критическая"`.
* Иначе → вернуть `"Некритическая"`.

---

## Главный процесс должен:

1. Создать пустой список `severity_check_list`.
2. В цикле:

   * Вызвать `CheckBugSeverity`;
   * Сформировать словарь:

```json
{
  "bug_id": "BUG-2024-001",
  "severity_status": "Критическая"
}
```

* Добавить словарь в `severity_check_list`.

3. Сохранить `severity_check_list` в файл
   `severity_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку критичности ошибки.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт эффективности исправления и формирование итогового отчёта

### Ситуация

После проверки критичности руководство решило автоматизировать оценку эффективности работы команды по исправлению ошибок.

Теперь требуется не только зафиксировать критичность, но и рассчитать показатель эффективности исправления на основе времени устранения и критичности, а также присвоить ошибке статус качества для включения в релизный отчёт.

Руководству разработки нужен структурированный список ошибок с расчётными данными для анализа качества релиза.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 ошибок.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateFixEfficiency`

**Параметр:**

* `bug_dict`

**Логика:**

* Получить `severity` и `fix_time_hours`.
* Базовый коэффициент критичности:
  - "critical" → 4
  - "high" → 3
  - "medium" → 2
  - "low" → 1
* Рассчитать эффективность:
  `коэффициент_критичности * 10 / fix_time_hours`
* Вернуть число (показатель эффективности).

---

### 2️⃣ Функция `DetermineQualityStatus`

**Параметр:**

* `efficiency_value`

**Логика:**

* Если `efficiency_value` ≥ 5 → вернуть `"Отличное качество"`
* Если `efficiency_value` ≥ 2 → вернуть `"Приемлемое качество"`
* Иначе → вернуть `"Требует улучшения"`

---

### 3️⃣ Функция `BuildBugReportRecord`

**Параметры:**

* `bug_id`
* `quality_status`
* `efficiency_score`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "bug_id": "BUG-2024-001",
  "quality_status": "Отличное качество",
  "efficiency_score": 5.0
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_bug_report_list`.
2. В цикле для каждой ошибки:

   * Вызвать `CheckBugSeverity` (проверить статус критичности);
   * Вызвать `CalculateFixEfficiency`;
   * Передать результат в `DetermineQualityStatus`;
   * Вызвать `BuildBugReportRecord`;
   * Добавить возвращённый словарь в `final_bug_report_list`.
3. Сохранить `final_bug_report_list` в файл
   `bug_report_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
