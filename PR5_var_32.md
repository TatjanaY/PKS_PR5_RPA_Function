# Вариант 32

## Кейс: Автоматизация приоритизации инцидентов в Service Desk

В крупной IT-компании внедряется система автоматической приоритизации инцидентов, поступающих в службу технической поддержки.
Руководство приняло решение исключить ручную классификацию заявок и минимизировать риски несвоевременного реагирования на критические сбои.

Ранее специалисты поддержки вручную оценивали влияние инцидента, срочность восстановления и количество затронутых пользователей, присваивая приоритеты в тикет-системе. Процесс занимал значительное время при массовом поступлении заявок и сопровождался риском несогласованности в оценке критичности.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел технической поддержки формирует список инцидентов с данными о характеристиках и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список инцидентов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "incident_id": "INC-2024-001",
  "impact_level": 3,
  "urgency_level": 2,
  "affected_users": 50
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка минимального порога влияния

### Ситуация

На первом этапе система должна проверить, достигает ли инцидент минимального порога влияния для регистрации в системе приоритизации.
Инциденты с уровнем влияния (`impact_level`) менее 2 считаются незначительными и не требуют автоматической обработки.

Отделу поддержки необходимо получить отдельный файл со статусом регистрации по каждому инциденту.

---

## Требуется:

1. Считать JSON-файл (6 инцидентов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckImpactEligibility`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `incident_dict`

**Логика:**

* Получить значение `impact_level` из словаря.
* Если `impact_level` < 2 → вернуть `"Не регистрируется"`.
* Если `impact_level` ≥ 2 → вернуть `"Зарегистрирован"`.

---

## Главный процесс должен:

1. Создать пустой список `impact_check_list`.
2. В цикле:

   * Вызвать `CheckImpactEligibility`;
   * Сформировать словарь:

```json
{
  "incident_id": "INC-2024-001",
  "registration_status": "Зарегистрирован"
}
```

* Добавить словарь в `impact_check_list`.

3. Сохранить `impact_check_list` в файл
   `impact_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку порога влияния.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт приоритета инцидента и формирование маршрута обработки

### Ситуация

После проверки регистрации руководство решило автоматизировать расчёт приоритета инцидентов для оптимального распределения нагрузки на специалистов.

Теперь требуется не только подтвердить регистрацию, но и рассчитать интегральный приоритет на основе уровня влияния, срочности и количества затронутых пользователей, а также присвоить инциденту маршрут эскалации.

Руководству поддержки нужен структурированный список инцидентов с расчётными данными для оперативного распределения заявок между командами.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 инцидентов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateIncidentPriority`

**Параметр:**

* `incident_dict`

**Логика:**

* Получить `impact_level`, `urgency_level`, `affected_users`.
* Рассчитать приоритет по формуле:
  `(impact_level * urgency_level * 2) + (affected_users / 25)`
* Вернуть число (интегральный приоритет).

---

### 2️⃣ Функция `DetermineEscalationRoute`

**Параметр:**

* `priority_value`

**Логика:**

* Если `priority_value` ≥ 15 → вернуть `"Критический: эскалация руководителю"`
* Если `priority_value` ≥ 8 → вернуть `"Высокий: старший инженер"`
* Если `priority_value` ≥ 4 → вернуть `"Средний: линейная поддержка"`
* Иначе → вернуть `"Низкий: очередь обработки"`

---

### 3️⃣ Функция `BuildIncidentRecord`

**Параметры:**

* `incident_id`
* `escalation_route`
* `priority_score`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "incident_id": "INC-2024-001",
  "escalation_route": "Высокий: старший инженер",
  "priority_score": 10.0
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_incident_list`.
2. В цикле для каждого инцидента:

   * Вызвать `CheckImpactEligibility` (проверить статус регистрации);
   * Вызвать `CalculateIncidentPriority`;
   * Передать результат в `DetermineEscalationRoute`;
   * Вызвать `BuildIncidentRecord`;
   * Добавить возвращённый словарь в `final_incident_list`.
3. Сохранить `final_incident_list` в файл
   `incident_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
