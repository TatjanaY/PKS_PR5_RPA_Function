# Вариант 19

## Кейс: Автоматизация контроля остатков на складе

В крупной логистической компании внедряется система автоматического мониторинга складских остатков для своевременного пополнения товарных запасов.
Руководство приняло решение исключить ручную сверку данных из учётной системы и минимизировать риски возникновения дефицита или излишков товаров.

Ранее сотрудники склада вручную анализировали текущие остатки, сравнивали их с минимальными порогами и формировали заявки на закупку. Процесс занимал значительное время и сопровождался риском ошибок при расчёте точек заказа.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Складской отдел формирует список товаров с данными об остатках и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список товаров;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "product_name": "Товар А",
  "current_stock": 25,
  "min_stock_level": 50,
  "daily_sales_avg": 10
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка достаточности складского остатка

### Ситуация

На первом этапе система должна проверить, не опустился ли текущий остаток товара ниже минимального порога.
Товары с остатком ниже минимального уровня требуют срочного пополнения.

Складскому отделу необходимо получить отдельный файл со статусом достаточности остатка по каждому товару.

---

## Требуется:

1. Считать JSON-файл (6 товаров).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckStockSufficiency`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `product_dict`

**Логика:**

* Получить значения `current_stock` и `min_stock_level` из словаря.
* Если `current_stock` < `min_stock_level` → вернуть `"Требуется пополнение"`.
* Если `current_stock` ≥ `min_stock_level` → вернуть `"Остаток достаточен"`.

---

## Главный процесс должен:

1. Создать пустой список `stock_check_list`.
2. В цикле:

   * Вызвать `CheckStockSufficiency`;
   * Сформировать словарь:

```json
{
  "product_name": "Товар А",
  "stock_status": "Требуется пополнение"
}
```

* Добавить словарь в `stock_check_list`.

3. Сохранить `stock_check_list` в файл
   `stock_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку достаточности остатка.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт объёма заказа и формирование заявки на закупку

### Ситуация

После проверки остатков руководство решило автоматизировать расчёт оптимального объёма заказа.

Теперь требуется не только выявить товары, требующие пополнения, но и рассчитать рекомендуемый объём заказа на основе средних продаж и страхового запаса, а также присвоить приоритет закупки.

Отделу закупок нужен структурированный список товаров с рекомендациями для формирования заказов поставщикам.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 товаров.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateOrderQuantity`

**Параметр:**

* `product_dict`

**Логика:**

* Получить `current_stock`, `min_stock_level`, `daily_sales_avg`.
* Рассчитать дефицит: `min_stock_level - current_stock`.
* Если дефицит ≤ 0 → вернуть 0.
* Рассчитать заказ: `дефицит + (daily_sales_avg * 7)` (недельный запас).
* Вернуть число.

---

### 2️⃣ Функция `DetermineOrderPriority`

**Параметр:**

* `order_quantity`

**Логика:**

* Если `order_quantity` = 0 → вернуть `"Заказ не требуется"`.
* Если `order_quantity` ≥ 100 → вернуть `"Высокий приоритет"`.
* Если `order_quantity` ≥ 50 → вернуть `"Средний приоритет"`.
* Иначе → вернуть `"Низкий приоритет"`.

---

### 3️⃣ Функция `BuildOrderRecord`

**Параметры:**

* `product_name`
* `priority_text`
* `order_quantity`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "product_name": "Товар А",
  "order_priority": "Высокий приоритет",
  "recommended_quantity": 125
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_order_list`.
2. В цикле для каждого товара:

   * Вызвать `CalculateOrderQuantity`;
   * Передать результат в `DetermineOrderPriority`;
   * Вызвать `BuildOrderRecord`;
   * Добавить возвращённый словарь в `final_order_list`.
3. Сохранить `final_order_list` в файл
   `order_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
