# Вариант 23

## Кейс: Автоматизация расчёта оптимального объёма закупки товаров

В крупной торговой компании внедряется система автоматического расчёта оптимальных объёмов закупки товаров у поставщиков.
Руководство приняло решение исключить ручные расчёты потребностей в товарах и минимизировать риски излишних запасов или дефицита на складе.

Ранее сотрудники отдела закупок вручную анализировали продажи, сезонность, текущие остатки и сроки поставки для определения объёмов заказа. Процесс занимал значительное время перед каждым заказом и сопровождался риском ошибок при прогнозировании спроса.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел закупок формирует список товаров с данными о продажах и запасах и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список товаров;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "product_code": "ART-001",
  "current_stock": 100,
  "avg_daily_sales": 15,
  "lead_time_days": 7,
  "safety_stock": 50
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка необходимости закупки товара

### Ситуация

На первом этапе система должна проверить, требуется ли закупка товара на основе текущих остатков и прогнозируемого спроса.
Товары, у которых текущий остаток покрывает спрос на период поставки, не требуют срочной закупки.

Отделу закупок необходимо получить отдельный файл со статусом необходимости закупки по каждому товару.

---

## Требуется:

1. Считать JSON-файл (6 товаров).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckPurchaseNeeded`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `product_dict`

**Логика:**

* Получить значения `current_stock`, `avg_daily_sales`, `lead_time_days` из словаря.
* Рассчитать потребность на период поставки: `avg_daily_sales * lead_time_days`.
* Если `current_stock` < потребность → вернуть `"Закупка требуется"`.
* Если `current_stock` ≥ потребность → вернуть `"Закупка не требуется"`.

---

## Главный процесс должен:

1. Создать пустой список `purchase_check_list`.
2. В цикле:

   * Вызвать `CheckPurchaseNeeded`;
   * Сформировать словарь:

```json
{
  "product_code": "ART-001",
  "purchase_status": "Закупка требуется"
}
```

* Добавить словарь в `purchase_check_list`.

3. Сохранить `purchase_check_list` в файл
   `purchase_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку необходимости закупки.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт оптимального объёма заказа и приоритета поставки

### Ситуация

После проверки необходимости закупки руководство решило усложнить процесс планирования заказов.

Теперь требуется рассчитать оптимальный объём заказа с учётом страхового запаса и срока поставки, а также присвоить приоритет заказа для планирования бюджета закупок.

Отделу закупок нужен структурированный список товаров с рекомендуемыми объёмами для формирования заказов поставщикам.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 товаров.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateOptimalOrderQuantity`

**Параметр:**

* `product_dict`

**Логика:**

* Получить `current_stock`, `avg_daily_sales`, `lead_time_days`, `safety_stock`.
* Рассчитать потребность на период поставки: `avg_daily_sales * lead_time_days`.
* Рассчитать целевой запас: `потребность + safety_stock`.
* Рассчитать объём заказа: `целевой запас - current_stock`.
* Если объём заказа ≤ 0 → вернуть 0.
* Вернуть число.

---

### 2️⃣ Функция `DetermineOrderPriority`

**Параметр:**

* `order_quantity`

**Логика:**

* Если `order_quantity` = 0 → вернуть `"Приоритет не требуется"`.
* Если `order_quantity` ≥ 200 → вернуть `"Высокий приоритет"`.
* Если `order_quantity` ≥ 100 → вернуть `"Средний приоритет"`.
* Иначе → вернуть `"Низкий приоритет"`.

---

### 3️⃣ Функция `BuildPurchaseRecord`

**Параметры:**

* `product_code`
* `priority_text`
* `order_quantity`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "product_code": "ART-001",
  "order_priority": "Высокий приоритет",
  "optimal_quantity": 250
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_purchase_list`.
2. В цикле для каждого товара:

   * Вызвать `CalculateOptimalOrderQuantity`;
   * Передать результат в `DetermineOrderPriority`;
   * Вызвать `BuildPurchaseRecord`;
   * Добавить возвращённый словарь в `final_purchase_list`.
3. Сохранить `final_purchase_list` в файл
   `purchase_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
