# Вариант 33

## Кейс: Автоматизация анализа времени реакции на заявки в Service Desk

В крупной IT-компании внедряется система автоматического мониторинга времени реакции специалистов службы технической поддержки на входящие заявки.
Руководство приняло решение исключить ручную выгрузку метрик из системы учёта и минимизировать риски нарушения регламентов обслуживания клиентов.

Ранее руководители службы поддержки вручную анализировали логи заявок, рассчитывали среднее время реакции и выявляли случаи нарушения нормативов. Процесс занимал значительное время при подготовке отчётов и сопровождался риском ошибок при агрегации данных.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Служба поддержки формирует список заявок с данными о времени обработки и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список заявок;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "ticket_id": "TKT-101",
  "created_time": "2024-01-15T09:00:00",
  "first_response_time": "2024-01-15T09:45:00",
  "priority": "high",
  "sla_limit_minutes": 60
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка соблюдения норматива времени реакции

### Ситуация

На первом этапе система должна проверить, уложился ли специалист в норматив времени реакции на заявку согласно приоритету обращения.
Заявки, по которым время реакции превысило установленный лимит, считаются нарушающими регламент.

Службе поддержки необходимо получить отдельный файл со статусом соблюдения норматива по каждой заявке.

---

## Требуется:

1. Считать JSON-файл (6 заявок).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckResponseTimeCompliance`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `ticket_dict`

**Логика:**

* Рассчитать время реакции в минутах (для упрощения: разница между `first_response_time` и `created_time` в минутах).
* Получить значение `sla_limit_minutes` из словаря.
* Если время реакции > `sla_limit_minutes` → вернуть `"Нарушение регламента"`.
* Если время реакции ≤ `sla_limit_minutes` → вернуть `"В рамках регламента"`.

*(Для упрощения: можно передать уже рассчитанное `response_minutes` в исходных данных)*

---

## Главный процесс должен:

1. Создать пустой список `compliance_check_list`.
2. В цикле:

   * Вызвать `CheckResponseTimeCompliance`;
   * Сформировать словарь:

```json
{
  "ticket_id": "TKT-101",
  "compliance_status": "В рамках регламента"
}
```

* Добавить словарь в `compliance_check_list`.

3. Сохранить `compliance_check_list` в файл
   `compliance_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку соблюдения норматива времени.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт индекса качества реакции и присвоение оценки специалисту

### Ситуация

После проверки соблюдения регламента руководство решило усложнить процесс оценки работы специалистов.

Теперь требуется рассчитать индекс качества реакции на основе фактического времени, приоритета заявки и отклонения от норматива, а также присвоить оценку работе специалиста для формирования персональной статистики.

Руководству службы поддержки нужен структурированный список заявок с индексами качества для анализа эффективности команды.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 заявок.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateResponseQualityIndex`

**Параметр:**

* `ticket_dict`

**Логика:**

* Получить `response_minutes`, `sla_limit_minutes`, `priority`.
* Если `response_minutes` = 0 → вернуть 100 (идеально).
* Рассчитать отклонение: `(response_minutes / sla_limit_minutes) * 100`.
* Базовый индекс = 100.
* Если отклонение > 100 → вычесть `(отклонение - 100) / 2` баллов.
* Если `priority` = "critical" и отклонение > 80 → вычесть дополнительные 10 баллов.
* Вернуть число (не менее 0, не более 100).

---

### 2️⃣ Функция `DeterminePerformanceRating`

**Параметр:**

* `quality_index`

**Логика:**

* Если `quality_index` ≥ 90 → вернуть `"Отлично"`.
* Если `quality_index` ≥ 70 → вернуть `"Удовлетворительно"`.
* Иначе → вернуть `"Требует улучшения"`.

---

### 3️⃣ Функция `BuildQualityRecord`

**Параметры:**

* `ticket_id`
* `performance_rating`
* `quality_index`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "ticket_id": "TKT-101",
  "performance_rating": "Отлично",
  "quality_index": 95
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_quality_list`.
2. В цикле для каждой заявки:

   * Вызвать `CalculateResponseQualityIndex`;
   * Передать результат в `DeterminePerformanceRating`;
   * Вызвать `BuildQualityRecord`;
   * Добавить возвращённый словарь в `final_quality_list`.
3. Сохранить `final_quality_list` в файл
   `quality_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
