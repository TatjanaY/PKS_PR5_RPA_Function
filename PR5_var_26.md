# Вариант 26

## Кейс: Автоматизация проверки допуска студентов к экзамену

В крупном учебном заведении внедряется система автоматической проверки допуска студентов к итоговой аттестации.
Руководство приняло решение исключить ручную сверку посещаемости и успеваемости, минимизировав риски допуска к экзаменам студентов, не выполнивших учебные требования.

Ранее методисты вручную анализировали журналы посещаемости, ведомости с оценками и отчёты о выполнении лабораторных работ. Процесс занимал значительное время перед экзаменационной сессией и сопровождался риском ошибок при обработке больших массивов данных.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Учебный отдел формирует список студентов с данными об успеваемости и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список студентов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "student_name": "Петрова П.П.",
  "attendance_percent": 85,
  "average_grade": 4.2,
  "labs_completed": 8
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка минимального порога посещаемости

### Ситуация

На первом этапе система должна проверить, достигает ли студент минимального порога посещаемости для допуска к экзамену.
Согласно учебному регламенту, минимальный порог посещаемости составляет 75%.

Учебному отделу необходимо получить отдельный файл со статусом допуска по посещаемости для каждого студента.

---

## Требуется:

1. Считать JSON-файл (6 студентов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckAttendanceEligibility`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `student_dict`

**Логика:**

* Получить значение `attendance_percent` из словаря.
* Если `attendance_percent` < 75 → вернуть `"Не допущен"`.
* Если `attendance_percent` ≥ 75 → вернуть `"Допущен"`.

---

## Главный процесс должен:

1. Создать пустой список `attendance_result_list`.
2. В цикле:

   * Вызвать `CheckAttendanceEligibility`;
   * Сформировать словарь:

```json
{
  "student_name": "Петрова П.П.",
  "attendance_status": "Допущен"
}
```

* Добавить словарь в `attendance_result_list`.

3. Сохранить `attendance_result_list` в файл
   `attendance_result.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку порога посещаемости.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Комплексная проверка допуска и формирование итогового решения

### Ситуация

После проверки посещаемости руководство решило автоматизировать комплексную оценку готовности студента к экзамену.

Теперь требуется не только подтвердить посещаемость, но и проверить средний балл и количество выполненных лабораторных работ, а также присвоить итоговый статус допуска для формирования экзаменационных ведомостей.

Деканату нужен структурированный список студентов с финальными статусами для организации экзаменационного процесса.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 студентов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateEligibilityScore`

**Параметр:**

* `student_dict`

**Логика:**

* Получить `average_grade` и `labs_completed`.
* Рассчитать балл готовности:
  `(average_grade * 2) + (labs_completed * 0.5)`
* Вернуть число (интегральный балл).

---

### 2️⃣ Функция `DetermineFinalAdmission`

**Параметры:**

* `attendance_ok` (булево)
* `eligibility_score` (число)

**Логика:**

* Если `attendance_ok` = `false` → вернуть `"Не допущен"`.
* Если `eligibility_score` ≥ 10 → вернуть `"Допущен с отличием"`.
* Если `eligibility_score` ≥ 7 → вернуть `"Допущен"`.
* Иначе → вернуть `"Условный допуск"`.

---

### 3️⃣ Функция `BuildAdmissionRecord`

**Параметры:**

* `student_name`
* `admission_status`
* `score_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "student_name": "Петрова П.П.",
  "final_admission": "Допущен с отличием",
  "eligibility_score": 12.4
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_admission_list`.
2. В цикле для каждого студента:

   * Вызвать `CheckAttendanceEligibility` (получить статус посещаемости);
   * Вызвать `CalculateEligibilityScore`;
   * Преобразовать статус посещаемости в булево значение;
   * Вызвать `DetermineFinalAdmission` с двумя параметрами;
   * Вызвать `BuildAdmissionRecord`;
   * Добавить возвращённый словарь в `final_admission_list`.
3. Сохранить `final_admission_list` в файл
   `admission_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
