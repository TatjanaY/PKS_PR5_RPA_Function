# Вариант 15

## Кейс: Автоматизация сегментации клиентов по уровню активности

В крупной торговой компании внедряется система автоматической сегментации клиентской базы для настройки таргетированных маркетинговых коммуникаций.
Руководство приняло решение исключить ручную выгрузку данных из CRM и минимизировать риски некорректного распределения клиентов по группам для рассылок.

Ранее маркетологи вручную анализировали даты последних покупок, частоту обращений и сумму затрат клиентов. Процесс занимал значительное время перед запуском кампаний и сопровождался риском ошибок при классификации статусов.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел маркетинга формирует список клиентов с данными об активности и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список клиентов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "client_name": "ЗАО «Актив»",
  "days_since_last_order": 45,
  "total_orders": 12,
  "total_spent": 500000.00
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка текущей активности клиента

### Ситуация

На первом этапе система должна проверить, является ли клиент активным на текущий момент.
Клиенты, у которых последняя покупка была более 90 дней назад, считаются неактивными.

Отделу маркетинга необходимо получить отдельный файл со статусом активности по каждому клиенту.

---

## Требуется:

1. Считать JSON-файл (6 клиентов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckClientActivity`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `client_dict`

**Логика:**

* Получить значение `days_since_last_order` из словаря.
* Если `days_since_last_order` > 90 → вернуть `"Неактивен"`.
* Если `days_since_last_order` ≤ 90 → вернуть `"Активен"`.

---

## Главный процесс должен:

1. Создать пустой список `activity_check_list`.
2. В цикле:

   * Вызвать `CheckClientActivity`;
   * Сформировать словарь:

```json
{
  "client_name": "ЗАО «Актив»",
  "activity_status": "Активен"
}
```

* Добавить словарь в `activity_check_list`.

3. Сохранить `activity_check_list` в файл
   `activity_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку статуса активности.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт балла лояльности и присвоение сегмента

### Ситуация

После проверки активности руководство решило усложнить процесс сегментации.

Теперь требуется рассчитать комплексный балл лояльности на основе давности заказов, их количества и суммы затрат, а также присвоить клиенту маркетинговый сегмент для настройки коммуникации.

Отделу маркетинга нужен структурированный список клиентов с сегментами для загрузки в рассылочный сервис.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 клиентов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateLoyaltyScore`

**Параметр:**

* `client_dict`

**Логика:**

* Получить `days_since_last_order`, `total_orders`, `total_spent`.
* Базовый балл = 0.
* Если `days_since_last_order` ≤ 30 → добавить 50 баллов.
* Если `days_since_last_order` ≤ 90 → добавить 20 баллов.
* Добавить `total_orders` * 5 баллов.
* Если `total_spent` > 1000000 → добавить 100 баллов.
* Вернуть итоговый балл.

---

### 2️⃣ Функция `DetermineClientSegment`

**Параметр:**

* `score_value`

**Логика:**

* Если `score_value` ≥ 200 → вернуть `"Сегмент VIP"`
* Если `score_value` ≥ 100 → вернуть `"Сегмент Стандарт"`
* Иначе → вернуть `"Сегмент Потенциальный"`

---

### 3️⃣ Функция `BuildSegmentRecord`

**Параметры:**

* `client_name`
* `segment_text`
* `loyalty_score`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "client_name": "ЗАО «Актив»",
  "segment": "Сегмент VIP",
  "loyalty_score": 250
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_segment_list`.
2. В цикле для каждого клиента:

   * Вызвать `CalculateLoyaltyScore`;
   * Передать результат в `DetermineClientSegment`;
   * Вызвать `BuildSegmentRecord`;
   * Добавить возвращённый словарь в `final_segment_list`.
3. Сохранить `final_segment_list` в файл
   `segment_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
