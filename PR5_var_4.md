# Вариант 4

## Кейс: Автоматизация проверки прохождения обязательного обучения

В крупной компании внедряется система автоматического контроля прохождения сотрудниками обязательных обучающих курсов.
Руководство приняло решение исключить ручную сверку отчётов об обучении и минимизировать риски допуска к работе сотрудников без необходимых сертификатов.

Ранее HR-специалисты вручную проверяли списки прошедших обучение, сверяли даты окончания курсов и сроки действия сертификатов. Процесс занимал значительное время и сопровождался риском пропуска просроченных документов.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

HR-отдел формирует список сотрудников с данными об обучении и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список сотрудников;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "name": "Волков В.В.",
  "course_completed": true,
  "certificate_valid_until": "2025-12-31"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка факта прохождения обучения

### Ситуация

На первом этапе система должна проверить, прошёл ли сотрудник обязательный курс обучения.
Допуск к работе возможен только при подтверждённом факте завершения курса.

HR-отделу необходимо получить отдельный файл со статусом прохождения обучения по каждому сотруднику.

---

## Требуется:

1. Считать JSON-файл (6 сотрудников).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckTrainingCompletion`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `employee_dict`

**Логика:**

* Получить значение `course_completed` из словаря.
* Если `course_completed` = `false` → вернуть `"Не прошёл"`.
* Если `course_completed` = `true` → вернуть `"Прошёл"`.

---

## Главный процесс должен:

1. Создать пустой список `training_result_list`.
2. В цикле:

   * Вызвать `CheckTrainingCompletion`;
   * Сформировать словарь:

```json
{
  "name": "Волков В.В.",
  "training_status": "Прошёл"
}
```

* Добавить словарь в `training_result_list`.

3. Сохранить `training_result_list` в файл
   `training_result.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку факта прохождения курса.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Проверка актуальности сертификата и формирование итогового статуса

### Ситуация

После проверки факта прохождения обучения руководство решило усложнить процесс контроля.

Теперь требуется также проверить срок действия сертификата и сформировать итоговый статус допуска сотрудника к работе.

Отделу комплаенса нужен структурированный список сотрудников с финальными статусами для автоматического обновления доступов в корпоративных системах.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 сотрудников.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CheckCertificateValidity`

**Параметр:**

* `employee_dict`

**Логика:**

* Получить значение `certificate_valid_until`.
* Сравнить дату с текущей (условно: если строка содержит год < 2024 → просрочен).
* Если сертификат просрочен → вернуть `false`.
* Если сертификат действителен → вернуть `true`.

*(Для упрощения: считать, что текущая дата — 2024-01-01; если год в дате < 2024 → просрочен)*

---

### 2️⃣ Функция `DetermineAccessStatus`

**Параметры:**

* `training_passed` (булево)
* `certificate_valid` (булево)

**Логика:**

* Если `training_passed` = `true` И `certificate_valid` = `true` → вернуть `"Допущен"`
* Иначе → вернуть `"Не допущен"`

---

### 3️⃣ Функция `BuildAccessRecord`

**Параметры:**

* `employee_name`
* `access_status`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "name": "Волков В.В.",
  "access_status": "Допущен"
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_access_list`.
2. В цикле для каждого сотрудника:

   * Проверить `course_completed` (из исходных данных);
   * Вызвать `CheckCertificateValidity`;
   * Вызвать `DetermineAccessStatus` с двумя булевыми параметрами;
   * Вызвать `BuildAccessRecord`;
   * Добавить возвращённый словарь в `final_access_list`.
3. Сохранить `final_access_list` в файл
   `access_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
