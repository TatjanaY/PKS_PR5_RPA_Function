# Вариант 8

## Кейс: Автоматизация контроля лимитов бюджета подразделений

В крупной компании внедряется система автоматического мониторинга расходования бюджетов структурными подразделениями.
Руководство приняло решение исключить ручную сверку фактических расходов с утверждёнными лимитами и минимизировать риски превышения бюджетных назначений.

Ранее финансовые специалисты вручную агрегировали данные из разных систем, сравнивали суммы с плановыми показателями и готовили отчёты для руководства. Процесс занимал значительное время в конце отчётного периода и сопровождался риском ошибок при консолидации данных.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Финансовый отдел формирует список подразделений с данными о расходах и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список подразделений;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "department": "Отдел маркетинга",
  "budget_limit": 500000,
  "actual_spent": 420000
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка соблюдения бюджетного лимита

### Ситуация

На первом этапе система должна проверить, не превысило ли подразделение утверждённый лимит расходов.
Подразделения, уложившиеся в лимит, получают положительный статус для дальнейшего планирования.

Финансовому отделу необходимо получить отдельный файл со статусом соблюдения лимита по каждому подразделению.

---

## Требуется:

1. Считать JSON-файл (6 подразделений).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckBudgetCompliance`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `department_dict`

**Логика:**

* Получить значения `budget_limit` и `actual_spent` из словаря.
* Если `actual_spent` > `budget_limit` → вернуть `"Превышение"`.
* Если `actual_spent` ≤ `budget_limit` → вернуть `"В рамках лимита"`.

---

## Главный процесс должен:

1. Создать пустой список `compliance_check_list`.
2. В цикле:

   * Вызвать `CheckBudgetCompliance`;
   * Сформировать словарь:

```json
{
  "department": "Отдел маркетинга",
  "compliance_status": "В рамках лимита"
}
```

* Добавить словарь в `compliance_check_list`.

3. Сохранить `compliance_check_list` в файл
   `compliance_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку соблюдения лимита.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт остатка бюджета и формирование рекомендаций

### Ситуация

После проверки соблюдения лимита руководство решило автоматизировать формирование рекомендаций по использованию остатка бюджета.

Теперь требуется не только зафиксировать факт превышения/соблюдения, но и рассчитать остаток средств, а также присвоить рекомендацию по дальнейшему расходованию.

Финансовому отделу нужен структурированный список подразделений с расчётными данными для планирования следующего отчётного периода.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 подразделений.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateBudgetRemainder`

**Параметр:**

* `department_dict`

**Логика:**

* Получить `budget_limit` и `actual_spent`.
* Рассчитать остаток:
  `budget_limit - actual_spent`
* Вернуть число (может быть отрицательным при превышении).

---

### 2️⃣ Функция `DetermineSpendingRecommendation`

**Параметр:**

* `remainder_value`

**Логика:**

* Если `remainder_value` ≥ 100000 → вернуть `"Рекомендуется освоить остаток"`
* Если `remainder_value` ≥ 0 → вернуть `"Расходование в норме"`
* Иначе → вернуть `"Требуется корректировка бюджета"`

---

### 3️⃣ Функция `BuildBudgetRecord`

**Параметры:**

* `department_name`
* `recommendation_text`
* `remainder_amount`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "department": "Отдел маркетинга",
  "recommendation": "Расходование в норме",
  "budget_remainder": 80000
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_budget_list`.
2. В цикле для каждого подразделения:

   * Вызвать `CalculateBudgetRemainder`;
   * Передать результат в `DetermineSpendingRecommendation`;
   * Вызвать `BuildBudgetRecord`;
   * Добавить возвращённый словарь в `final_budget_list`.
3. Сохранить `final_budget_list` в файл
   `budget_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
