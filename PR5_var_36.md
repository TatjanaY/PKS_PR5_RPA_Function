# Вариант 36

## Кейс: Автоматизация анализа активности пользователей корпоративной системы

В крупной компании внедряется система автоматического мониторинга активности сотрудников в корпоративных цифровых сервисах.
Руководство приняло решение исключить ручную оценку вовлечённости персонала в использование внутренних инструментов и минимизировать риски принятия кадровых решений на основе неполных данных.

Ранее HR-аналитики вручную агрегировали логи входов в систему, время сессий и количество выполненных действий из разных источников, рассчитывали показатели вовлечённости в таблицах. Процесс занимал значительное время при подготовке отчётности и сопровождался риском несогласованности методик расчёта.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

IT-отдел формирует список пользователей с данными об активности и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список пользователей;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "user_name": "Николаев Н.Н.",
  "logins_count": 45,
  "total_session_hours": 120,
  "actions_completed": 350
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка минимального порога входов в систему

### Ситуация

На первом этапе система должна проверить, достигает ли пользователь минимального порога количества входов в систему для учёта в аналитике вовлечённости.
Порог установлен на уровне 10 входов за отчётный период.

HR-отделу необходимо получить отдельный файл со статусом учёта активности по каждому пользователю.

---

## Требуется:

1. Считать JSON-файл (6 пользователей).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckLoginThreshold`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `user_dict`

**Логика:**

* Получить значение `logins_count` из словаря.
* Если `logins_count` < 10 → вернуть `"Не учитывается"`.
* Если `logins_count` ≥ 10 → вернуть `"Учитывается"`.

---

## Главный процесс должен:

1. Создать пустой список `login_check_list`.
2. В цикле:

   * Вызвать `CheckLoginThreshold`;
   * Сформировать словарь:

```json
{
  "user_name": "Николаев Н.Н.",
  "accounting_status": "Учитывается"
}
```

* Добавить словарь в `login_check_list`.

3. Сохранить `login_check_list` в файл
   `login_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку порога входов.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт интегрального балла вовлечённости и формирование категории

### Ситуация

После проверки порога входов руководство решило автоматизировать комплексную оценку вовлечённости пользователей.

Теперь требуется не только подтвердить учёт активности, но и рассчитать интегральный балл на основе трёх факторов (частота входов, длительность сессий, количество действий), а также присвоить пользователю категорию вовлечённости для HR-аналитики.

Отделу развития персонала нужен структурированный список пользователей с расчётными данными для планирования программ адаптации и обучения.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 пользователей.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateActivityScore`

**Параметр:**

* `user_dict`

**Логика:**

* Получить `logins_count`, `total_session_hours`, `actions_completed`.
* Рассчитать балл активности по формуле:
  `(logins_count * 1) + (total_session_hours * 2) + (actions_completed * 0.5)`
* Вернуть число (интегральный балл активности).

---

### 2️⃣ Функция `DetermineEngagementLevel`

**Параметр:**

* `activity_score`

**Логика:**

* Если `activity_score` ≥ 50 → вернуть `"Высокая вовлечённость"`
* Если `activity_score` ≥ 30 → вернуть `"Средняя вовлечённость"`
* Иначе → вернуть `"Низкая вовлечённость"`

---

### 3️⃣ Функция `BuildUserRecord`

**Параметры:**

* `user_name`
* `engagement_label`
* `score_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "user_name": "Николаев Н.Н.",
  "engagement_level": "Высокая вовлечённость",
  "activity_score": 285.0
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_user_list`.
2. В цикле для каждого пользователя:

   * Вызвать `CalculateActivityScore`;
   * Передать результат в `DetermineEngagementLevel`;
   * Вызвать `BuildUserRecord`;
   * Добавить возвращённый словарь в `final_user_list`.
3. Сохранить `final_user_list` в файл
   `engagement_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
