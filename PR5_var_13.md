# Вариант 13

## Кейс: Автоматизация оценки эффективности менеджеров по продажам

В крупной торговой компании внедряется система автоматической оценки эффективности работы менеджеров по продажам.
Руководство приняло решение исключить ручную сводку данных из CRM и минимизировать риски субъективной оценки при распределении бонусов.

Ранее руководители отдела продаж вручную собирали данные о выполнении планов, количестве новых клиентов и сумме сделок. Процесс занимал значительное время в конце месяца и сопровождался риском ошибок при подсчёте процентов выполнения.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел продаж формирует список менеджеров с показателями и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список менеджеров;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "name": "Смирнов А.А.",
  "sales_plan": 1000000.00,
  "sales_fact": 1200000.00,
  "new_clients": 5
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка выполнения плана продаж

### Ситуация

На первом этапе система должна проверить, выполнил ли менеджер минимальный план продаж.
Менеджеры, не выполнившие план, не допускаются к участию в конкурсе на звание «Лучший сотрудник».

Отделу продаж необходимо получить отдельный файл со статусом выполнения плана по каждому менеджеру.

---

## Требуется:

1. Считать JSON-файл (6 менеджеров).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckPlanFulfillment`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `manager_dict`

**Логика:**

* Получить значения `sales_plan` и `sales_fact` из словаря.
* Если `sales_fact` < `sales_plan` → вернуть `"План не выполнен"`.
* Если `sales_fact` ≥ `sales_plan` → вернуть `"План выполнен"`.

---

## Главный процесс должен:

1. Создать пустой список `plan_check_list`.
2. В цикле:

   * Вызвать `CheckPlanFulfillment`;
   * Сформировать словарь:

```json
{
  "name": "Смирнов А.А.",
  "plan_status": "План выполнен"
}
```

* Добавить словарь в `plan_check_list`.

3. Сохранить `plan_check_list` в файл
   `plan_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку выполнения плана.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт процента выполнения и присвоение грейда

### Ситуация

После проверки факта выполнения плана руководство решило усложнить систему оценки.

Теперь требуется рассчитать точный процент выполнения плана и присвоить менеджеру квалификационный грейд (Уровень) для начисления повышенных коэффициентов премии.

Отделу мотивации нужен структурированный список менеджеров с грейдами для формирования премиального фонда.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 менеджеров.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateFulfillmentPercent`

**Параметр:**

* `manager_dict`

**Логика:**

* Получить `sales_plan` и `sales_fact`.
* Если `sales_plan` = 0 → вернуть 0.
* Рассчитать процент:
  `(sales_fact / sales_plan) * 100`
* Вернуть число.

---

### 2️⃣ Функция `DetermineManagerGrade`

**Параметр:**

* `percent_value`

**Логика:**

* Если `percent_value` ≥ 120 → вернуть `"Грейд A (Лидер)"`
* Если `percent_value` ≥ 100 → вернуть `"Грейд B (Стандарт)"`
* Иначе → вернуть `"Грейд C (На пересмотр)"`

---

### 3️⃣ Функция `BuildManagerRecord`

**Параметры:**

* `employee_name`
* `grade_text`
* `percent_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "name": "Смирнов А.А.",
  "grade": "Грейд A (Лидер)",
  "fulfillment_percent": 120.0
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_sales_list`.
2. В цикле для каждого менеджера:

   * Вызвать `CalculateFulfillmentPercent`;
   * Передать результат в `DetermineManagerGrade`;
   * Вызвать `BuildManagerRecord`;
   * Добавить возвращённый словарь в `final_sales_list`.
3. Сохранить `final_sales_list` в файл
   `sales_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
