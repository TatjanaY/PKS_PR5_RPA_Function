# Вариант 21

## Кейс: Автоматизация формирования списка заказов к отгрузке

В крупной логистической компании внедряется система автоматической подготовки заказов к отгрузке со склада.
Руководство приняло решение исключить ручную обработку заказов и минимизировать риски ошибок при комплектации и отправке товаров клиентам.

Ранее сотрудники склада вручную проверяли статусы заказов, сверяли адреса доставки и приоритеты отгрузки. Процесс занимал значительное время и сопровождался риском перепутывания заказов или отгрузки не в той последовательности.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел логистики формирует список заказов с данными об отгрузке и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список заказов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "order_id": "ORD-001",
  "client_name": "ООО «Партнёр»",
  "items_count": 15,
  "is_paid": true,
  "delivery_priority": "standard"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка оплаты заказа

### Ситуация

На первом этапе система должна проверить, оплачен ли заказ клиентом.
Заказы без подтверждения оплаты не могут быть переданы на отгрузку.

Отделу логистики необходимо получить отдельный файл со статусом оплаты по каждому заказу.

---

## Требуется:

1. Считать JSON-файл (6 заказов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckPaymentStatus`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `order_dict`

**Логика:**

* Получить значение `is_paid` из словаря.
* Если `is_paid` = `false` → вернуть `"Оплата не подтверждена"`.
* Если `is_paid` = `true` → вернуть `"Оплата подтверждена"`.

---

## Главный процесс должен:

1. Создать пустой список `payment_check_list`.
2. В цикле:

   * Вызвать `CheckPaymentStatus`;
   * Сформировать словарь:

```json
{
  "order_id": "ORD-001",
  "payment_status": "Оплата подтверждена"
}
```

* Добавить словарь в `payment_check_list`.

3. Сохранить `payment_check_list` в файл
   `payment_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку статуса оплаты.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Формирование очереди отгрузки с приоритетами

### Ситуация

После проверки оплаты руководство решило усложнить процесс подготовки отгрузок.

Теперь требуется рассчитать приоритет отгрузки на основе статуса оплаты, количества товаров в заказе и типа доставки, а также сформировать итоговую очередь отгрузки.

Отделу логистики нужен структурированный список заказов с приоритетами для планирования работы склада.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 заказов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateShipmentPriority`

**Параметр:**

* `order_dict`

**Логика:**

* Получить `is_paid`, `items_count`, `delivery_priority`.
* Если `is_paid` = `false` → вернуть 0 (не отгружается).
* Базовый приоритет = 10.
* Если `delivery_priority` = "express" → добавить 50 баллов.
* Если `delivery_priority` = "priority" → добавить 30 баллов.
* Если `items_count` > 20 → добавить 10 баллов.
* Вернуть итоговый приоритет.

---

### 2️⃣ Функция `DetermineShipmentQueue`

**Параметр:**

* `priority_value`

**Логика:**

* Если `priority_value` = 0 → вернуть `"В ожидании оплаты"`.
* Если `priority_value` ≥ 60 → вернуть `"Первая очередь"`.
* Если `priority_value` ≥ 40 → вернуть `"Вторая очередь"`.
* Иначе → вернуть `"Третья очередь"`.

---

### 3️⃣ Функция `BuildShipmentRecord`

**Параметры:**

* `order_id`
* `queue_text`
* `priority_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "order_id": "ORD-001",
  "shipment_queue": "Первая очередь",
  "priority_score": 70
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_shipment_list`.
2. В цикле для каждого заказа:

   * Вызвать `CalculateShipmentPriority`;
   * Передать результат в `DetermineShipmentQueue`;
   * Вызвать `BuildShipmentRecord`;
   * Добавить возвращённый словарь в `final_shipment_list`.
3. Сохранить `final_shipment_list` в файл
   `shipment_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
