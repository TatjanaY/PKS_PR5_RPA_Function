# Вариант 27

## Кейс: Автоматизация анализа посещаемости занятий

В крупном учебном заведении внедряется система автоматического мониторинга посещаемости студентов для своевременного выявления групп риска и принятия профилактических мер.
Руководство приняло решение исключить ручную обработку журналов посещаемости и минимизировать риски пропуска студентов с критически низким уровнем присутствия на занятиях.

Ранее кураторы групп вручную сводили данные из электронных журналов, рассчитывали проценты посещаемости и формировали списки студентов для бесед. Процесс занимал значительное время в середине семестра и сопровождался риском ошибок при подсчёте пропусков.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Учебный отдел формирует список студентов с данными о посещаемости и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список студентов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "student_name": "Петрова П.П.",
  "group": "ЭК-201",
  "total_lessons": 40,
  "attended_lessons": 32,
  "excused_absences": 3
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка минимального порога посещаемости

### Ситуация

На первом этапе система должна проверить, соблюдает ли студент минимальный порог посещаемости (80% от общего количества занятий).
Студенты с посещаемостью ниже порога требуют внимания куратора.

Учебному отделу необходимо получить отдельный файл со статусом посещаемости по каждому студенту.

---

## Требуется:

1. Считать JSON-файл (6 студентов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckAttendanceThreshold`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `student_dict`

**Логика:**

* Получить значения `total_lessons` и `attended_lessons` из словаря.
* Если `total_lessons` = 0 → вернуть `"Нет данных"`.
* Рассчитать процент: `(attended_lessons / total_lessons) * 100`.
* Если процент < 80 → вернуть `"Ниже порога"`.
* Если процент ≥ 80 → вернуть `"В норме"`.

---

## Главный процесс должен:

1. Создать пустой список `attendance_check_list`.
2. В цикле:

   * Вызвать `CheckAttendanceThreshold`;
   * Сформировать словарь:

```json
{
  "student_name": "Петрова П.П.",
  "attendance_status": "В норме"
}
```

* Добавить словарь в `attendance_check_list`.

3. Сохранить `attendance_check_list` в файл
   `attendance_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку порога посещаемости.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт индекса дисциплины и формирование рекомендаций

### Ситуация

После проверки порога посещаемости руководство решило усложнить процесс анализа.

Теперь требуется рассчитать комплексный индекс дисциплины с учётом уважительных пропусков и присвоить студенту рекомендацию для куратора.

Деканату нужен структурированный список студентов с индексами и рекомендациями для планирования индивидуальной работы.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 студентов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateDisciplineIndex`

**Параметр:**

* `student_dict`

**Логика:**

* Получить `total_lessons`, `attended_lessons`, `excused_absences`.
* Если `total_lessons` = 0 → вернуть 0.
* Рассчитать базовый процент: `(attended_lessons / total_lessons) * 100`.
* Добавить бонус за уважительные: `excused_absences * 2`.
* Вернуть число (максимум 100).

---

### 2️⃣ Функция `DetermineRecommendation`

**Параметр:**

* `index_value`

**Логика:**

* Если `index_value` ≥ 90 → вернуть `"Рекомендация: поощрить"`.
* Если `index_value` ≥ 80 → вернуть `"Рекомендация: наблюдать"`.
* Иначе → вернуть `"Рекомендация: провести беседу"`.

---

### 3️⃣ Функция `BuildAttendanceRecord`

**Параметры:**

* `student_name`
* `recommendation_text`
* `discipline_index`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "student_name": "Петрова П.П.",
  "recommendation": "Рекомендация: наблюдать",
  "discipline_index": 86
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_attendance_list`.
2. В цикле для каждого студента:

   * Вызвать `CalculateDisciplineIndex`;
   * Передать результат в `DetermineRecommendation`;
   * Вызвать `BuildAttendanceRecord`;
   * Добавить возвращённый словарь в `final_attendance_list`.
3. Сохранить `final_attendance_list` в файл
   `attendance_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
