# Вариант 17

## Кейс: Автоматизация анализа повторных покупок клиентов

В крупной розничной компании внедряется система автоматического анализа повторных покупок для повышения лояльности клиентов.
Руководство приняло решение исключить ручную выгрузку данных о покупках и минимизировать риски упущенной выгоды от работы с постоянными клиентами.

Ранее маркетологи вручную анализировали историю заказов, выявляли клиентов с повторными покупками и рассчитывали интервалы между заказами. Процесс занимал значительное время перед запуском программ лояльности и сопровождался риском ошибок при подсчёте частоты покупок.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел маркетинга формирует список клиентов с данными о покупках и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список клиентов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "client_id": "CL-001",
  "total_purchases": 8,
  "first_purchase_date": "2023-01-15",
  "last_purchase_date": "2024-11-20"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка факта повторной покупки

### Ситуация

На первом этапе система должна проверить, совершил ли клиент более одной покупки.
Клиенты с единичной покупкой не считаются повторными и требуют отдельной стратегии вовлечения.

Отделу маркетинга необходимо получить отдельный файл со статусом повторной покупки по каждому клиенту.

---

## Требуется:

1. Считать JSON-файл (6 клиентов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckRepeatPurchase`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `client_dict`

**Логика:**

* Получить значение `total_purchases` из словаря.
* Если `total_purchases` ≤ 1 → вернуть `"Первичный клиент"`.
* Если `total_purchases` > 1 → вернуть `"Повторный клиент"`.

---

## Главный процесс должен:

1. Создать пустой список `repeat_check_list`.
2. В цикле:

   * Вызвать `CheckRepeatPurchase`;
   * Сформировать словарь:

```json
{
  "client_id": "CL-001",
  "purchase_status": "Повторный клиент"
}
```

* Добавить словарь в `repeat_check_list`.

3. Сохранить `repeat_check_list` в файл
   `repeat_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку факта повторной покупки.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт частоты покупок и присвоение категории лояльности

### Ситуация

После проверки факта повторной покупки руководство решило усложнить процесс анализа.

Теперь требуется рассчитать среднюю частоту покупок и присвоить клиенту категорию лояльности для настройки персональных предложений.

Отделу маркетинга нужен структурированный список клиентов с категориями для интеграции с CRM-системой.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 клиентов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculatePurchaseFrequency`

**Параметр:**

* `client_dict`

**Логика:**

* Получить `total_purchases` и даты первой/последней покупки.
* Для упрощения: рассчитать `count_months` - количество месяцев между покупками (условно).
* Если `total_purchases` ≤ 1 → вернуть 0. 
* Рассчитать частоту:
  `total_purchases / count_months` (покупок в месяц, условно).
* Вернуть число.

---

### 2️⃣ Функция `DetermineLoyaltyCategory`

**Параметр:**

* `frequency_value`

**Логика:**

* Если `frequency_value` ≥ 1.0 → вернуть `"Категория Platinum"`
* Если `frequency_value` ≥ 0.5 → вернуть `"Категория Gold"`
* Иначе → вернуть `"Категория Silver"`

---

### 3️⃣ Функция `BuildLoyaltyRecord`

**Параметры:**

* `client_id`
* `loyalty_category`
* `frequency_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "client_id": "CL-001",
  "loyalty_category": "Категория Gold",
  "purchase_frequency": 0.67
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_loyalty_list`.
2. В цикле для каждого клиента:

   * Вызвать `CalculatePurchaseFrequency`;
   * Передать результат в `DetermineLoyaltyCategory`;
   * Вызвать `BuildLoyaltyRecord`;
   * Добавить возвращённый словарь в `final_loyalty_list`.
3. Сохранить `final_loyalty_list` в файл
   `loyalty_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
