# Вариант 9

## Кейс: Автоматизация анализа дебиторской задолженности клиентов

В крупной компании внедряется система автоматического мониторинга дебиторской задолженности клиентов.
Руководство приняло решение исключить ручную сверку платёжных дисциплин и минимизировать риски работы с проблемными контрагентами.

Ранее сотрудники финансового отдела вручную анализировали сроки оплаты, суммы задолженности и историю платежей клиентов. Процесс занимал значительное время и сопровождался риском пропуска просроченных обязательств.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Финансовый отдел формирует список клиентов с данными о задолженности и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список клиентов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "client_name": "ООО «Вектор»",
  "debt_amount": 250000.00,
  "days_overdue": 15,
  "payment_history": "regular"
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка наличия просроченной задолженности

### Ситуация

На первом этапе система должна проверить, имеет ли клиент просроченную задолженность.
Клиенты с задолженностью, просроченной более чем на 30 дней, требуют особого внимания.

Финансовому отделу необходимо получить отдельный файл со статусом просрочки по каждому клиенту.

---

## Требуется:

1. Считать JSON-файл (6 клиентов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckDebtOverdue`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `client_dict`

**Логика:**

* Получить значение `days_overdue` из словаря.
* Если `days_overdue` > 30 → вернуть `"Критическая просрочка"`.
* Если `days_overdue` > 0 → вернуть `"Текущая просрочка"`.
* Если `days_overdue` = 0 → вернуть `"Без просрочки"`.

---

## Главный процесс должен:

1. Создать пустой список `overdue_check_list`.
2. В цикле:

   * Вызвать `CheckDebtOverdue`;
   * Сформировать словарь:

```json
{
  "client_name": "ООО «Вектор»",
  "overdue_status": "Текущая просрочка"
}
```

* Добавить словарь в `overdue_check_list`.

3. Сохранить `overdue_check_list` в файл
   `overdue_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку статуса просрочки.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Формирование рейтинга надёжности клиентов

### Ситуация

После проверки просрочки руководство решило усложнить процесс оценки контрагентов.

Теперь требуется рассчитать комплексный рейтинг надёжности клиента на основе суммы задолженности, срока просрочки и истории платежей, а также присвоить категорию риска.

Отделу кредитного контроля нужен структурированный список клиентов с рейтингами для принятия решений о дальнейших отгрузках.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 клиентов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateReliabilityScore`

**Параметр:**

* `client_dict`

**Логика:**

* Получить `debt_amount`, `days_overdue` и `payment_history`.
* Базовый балл = 100.
* Если `days_overdue` > 30 → вычесть 40 баллов.
* Если `days_overdue` > 0 и ≤ 30 → вычесть 20 баллов.
* Если `payment_history` = "irregular" → вычесть 15 баллов.
* Если `debt_amount` > 500000 → вычесть 10 баллов.
* Вернуть итоговый балл (не менее 0).

---

### 2️⃣ Функция `DetermineRiskCategory`

**Параметр:**

* `score_value`

**Логика:**

* Если `score_value` ≥ 80 → вернуть `"Низкий риск"`
* Если `score_value` ≥ 50 → вернуть `"Средний риск"`
* Иначе → вернуть `"Высокий риск"`

---

### 3️⃣ Функция `BuildClientRecord`

**Параметры:**

* `client_name`
* `risk_category`
* `reliability_score`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "client_name": "ООО «Вектор»",
  "risk_category": "Средний риск",
  "reliability_score": 65
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_client_list`.
2. В цикле для каждого клиента:

   * Вызвать `CalculateReliabilityScore`;
   * Передать результат в `DetermineRiskCategory`;
   * Вызвать `BuildClientRecord`;
   * Добавить возвращённый словарь в `final_client_list`.
3. Сохранить `final_client_list` в файл
   `client_rating_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
