# Вариант 6

## Кейс: Автоматизация контроля переработок и сверхурочных часов

В крупной компании внедряется система автоматического мониторинга сверхурочной работы сотрудников.
Руководство приняло решение исключить ручную проверку табелей учёта рабочего времени и минимизировать риски нарушения трудового законодательства regarding overtime limits.

Ранее HR-специалисты вручную суммировали часы переработок, сверяли их с нормативами и рассчитывали компенсации. Процесс занимал значительное время в конце отчётного периода и сопровождался риском арифметических ошибок.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

HR-отдел формирует список сотрудников с данными о отработанных часах и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список сотрудников;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "name": "Зайцев З.З.",
  "normal_hours": 160,
  "overtime_hours": 25,
  "hourly_rate": 500
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка лимита сверхурочных часов

### Ситуация

На первом этапе система должна проверить, не превысил ли сотрудник установленный месячный лимит сверхурочных часов.
Согласно внутреннему регламенту, лимит составляет 20 часов в месяц.

HR-отделу необходимо получить отдельный файл со статусом соблюдения лимита по каждому сотруднику.

---

## Требуется:

1. Считать JSON-файл (6 сотрудников).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckOvertimeLimit`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `employee_dict`

**Логика:**

* Получить значение `overtime_hours` из словаря.
* Если `overtime_hours` > 20 → вернуть `"Превышение"`.
* Если `overtime_hours` ≤ 20 → вернуть `"В норме"`.

---

## Главный процесс должен:

1. Создать пустой список `overtime_check_list`.
2. В цикле:

   * Вызвать `CheckOvertimeLimit`;
   * Сформировать словарь:

```json
{
  "name": "Зайцев З.З.",
  "limit_status": "В норме"
}
```

* Добавить словарь в `overtime_check_list`.

3. Сохранить `overtime_check_list` в файл
   `overtime_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку лимита часов.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт компенсации и согласование выплаты

### Ситуация

После проверки лимита руководство решило автоматизировать расчёт компенсаций за сверхурочную работу.

Теперь требуется не только зафиксировать часы, но и рассчитать сумму оплаты (в двойном размере), а также проверить возможность выплаты в рамках бюджета подразделения.

Отделу расчёта заработной платы нужен структурированный список сотрудников с финальными суммами для проведения платежей.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 сотрудников.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateOvertimePay`

**Параметр:**

* `employee_dict`

**Логика:**

* Получить `overtime_hours` и `hourly_rate`.
* Рассчитать сумму (двойной тариф):
  `overtime_hours * hourly_rate * 2`
* Вернуть число.

---

### 2️⃣ Функция `DeterminePaymentApproval`

**Параметр:**

* `payment_amount`

**Логика:**

* Если `payment_amount` > 50000 → вернуть `"Требуется согласование"`
* Иначе → вернуть `"Согласовано автоматически"`

---

### 3️⃣ Функция `BuildPaymentRecord`

**Параметры:**

* `employee_name`
* `approval_status`
* `amount_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "name": "Зайцев З.З.",
  "payment_status": "Согласовано автоматически",
  "total_amount": 25000
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_payment_list`.
2. В цикле для каждого сотрудника:

   * Вызвать `CalculateOvertimePay`;
   * Передать результат в `DeterminePaymentApproval`;
   * Вызвать `BuildPaymentRecord`;
   * Добавить возвращённый словарь в `final_payment_list`.
3. Сохранить `final_payment_list` в файл
   `payment_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
