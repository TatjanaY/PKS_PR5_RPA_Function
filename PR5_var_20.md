# Вариант 20

## Кейс: Автоматизация проверки сроков годности товаров на складе

В крупной логистической компании внедряется система автоматического мониторинга сроков годности товарных позиций на складах.
Руководство приняло решение исключить ручную проверку партий товаров и минимизировать риски реализации просроченной продукции.

Ранее кладовщики вручную сверяли даты производства и сроки годности, формировали списки товаров, требующих уценки или списания. Процесс занимал значительное время при инвентаризации и сопровождался риском пропуска критических позиций.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Складской отдел формирует список товарных позиций с данными о сроках и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список товаров;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "product_name": "Молоко 3.2%",
  "production_date": "2024-01-15",
  "shelf_life_days": 14,
  "quantity": 120
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка актуальности срока годности

### Ситуация

На первом этапе система должна проверить, не истёк ли срок годности товарной позиции на текущую дату.
Для упрощения считаем, что текущая дата — 2024-01-25. Товар считается просроченным, если с даты производства прошло больше дней, чем указано в `shelf_life_days`.

Складскому отделу необходимо получить отдельный файл со статусом годности по каждой товарной позиции.

---

## Требуется:

1. Считать JSON-файл (6 товаров).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckProductValidity`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `product_dict`

**Логика:**

* Получить значения `production_date`, `shelf_life_days`.
* Рассчитать количество дней с даты производства (для упрощения: извлечь день из строки даты и сравнить).
* Если дней с производства > `shelf_life_days` → вернуть `"Просрочен"`.
* Иначе → вернуть `"Годен"`.

*(Для упрощения: если `production_date` содержит день < 15 при текущем дне 25 и сроке 14 дней → просрочен)*

---

## Главный процесс должен:

1. Создать пустой список `validity_check_list`.
2. В цикле:

   * Вызвать `CheckProductValidity`;
   * Сформировать словарь:

```json
{
  "product_name": "Молоко 3.2%",
  "validity_status": "Годен"
}
```

* Добавить словарь в `validity_check_list`.

3. Сохранить `validity_check_list` в файл
   `validity_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку срока годности.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Формирование рекомендаций по обработке товаров

### Ситуация

После проверки срока годности руководство решило автоматизировать формирование рекомендаций по дальнейшей обработке товарных позиций.

Теперь требуется не только зафиксировать статус годности, но и рассчитать остаточный срок хранения, а также присвоить приоритет обработки для оптимизации складских операций.

Отделу логистики нужен структурированный список товаров с рекомендациями для планирования отгрузок и уценок.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 товаров.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateRemainingShelfLife`

**Параметр:**

* `product_dict`

**Логика:**

* Получить `production_date` и `shelf_life_days`.
* Рассчитать прошедшие дни (упрощённо: 25 - день из `production_date`).
* Рассчитать остаток:
  `shelf_life_days - прошедшие_дни`
* Вернуть число (может быть отрицательным при просрочке).

---

### 2️⃣ Функция `DetermineHandlingPriority`

**Параметр:**

* `remaining_days`

**Логика:**

* Если `remaining_days` ≤ 0 → вернуть `"Срочное списание"`
* Если `remaining_days` ≤ 3 → вернуть `"Приоритетная отгрузка"`
* Если `remaining_days` ≤ 7 → вернуть `"Плановая реализация"`
* Иначе → вернуть `"Стандартное хранение"`

---

### 3️⃣ Функция `BuildProductRecord`

**Параметры:**

* `product_name`
* `priority_label`
* `remaining_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "product_name": "Молоко 3.2%",
  "handling_priority": "Плановая реализация",
  "remaining_days": 5
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_product_list`.
2. В цикле для каждого товара:

   * Вызвать `CalculateRemainingShelfLife`;
   * Передать результат в `DetermineHandlingPriority`;
   * Вызвать `BuildProductRecord`;
   * Добавить возвращённый словарь в `final_product_list`.
3. Сохранить `final_product_list` в файл
   `product_handling_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
