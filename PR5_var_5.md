# Вариант 5

## Кейс: Автоматизация формирования списка сотрудников на аттестацию

В крупной компании внедряется система автоматической подготовки списков сотрудников, подлежащих обязательной периодической аттестации.
Руководство приняло решение исключить ручную фильтрацию данных и минимизировать риски пропуска сотрудников, имеющих основания для внеочередной аттестации.

Ранее HR-специалисты вручную анализировали даты последней аттестации, результаты предыдущих оценок и статусы должностных перемещений. Процесс занимал значительное время и сопровождался риском ошибок при работе с большими массивами данных.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

HR-отдел формирует список сотрудников с данными об аттестации и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список сотрудников;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "name": "Морозов М.М.",
  "last_attestation_year": 2022,
  "previous_result": "Успешно",
  "position_changed": false
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка необходимости аттестации по сроку

### Ситуация

На первом этапе система должна проверить, прошло ли более 2 лет с момента последней аттестации сотрудника.
Сотрудники, не проходившие аттестацию более 2 лет, подлежат включению в список на текущий период.

HR-отделу необходимо получить отдельный файл со статусом необходимости аттестации по каждому сотруднику.

---

## Требуется:

1. Считать JSON-файл (6 сотрудников).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckAttestationDue`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `employee_dict`

**Логика:**

* Получить значение `last_attestation_year` из словаря.
* Если текущий год (условно 2024) - `last_attestation_year` > 2 → вернуть `"Требуется"`.
* Иначе → вернуть `"Не требуется"`.

---

## Главный процесс должен:

1. Создать пустой список `attestation_check_list`.
2. В цикле:

   * Вызвать `CheckAttestationDue`;
   * Сформировать словарь:

```json
{
  "name": "Морозов М.М.",
  "attestation_required": "Требуется"
}
```

* Добавить словарь в `attestation_check_list`.

3. Сохранить `attestation_check_list` в файл
   `attestation_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку срока аттестации.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Формирование итогового списка с приоритетами

### Ситуация

После проверки срока аттестации руководство решило усложнить процесс формирования списка.

Теперь требуется учитывать дополнительные факторы: результат предыдущей аттестации и факт смены должности. На основании комплексной оценки необходимо присвоить приоритет включения в список.

Отделу развития персонала нужен структурированный список сотрудников с приоритетами для планирования графика аттестации.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 сотрудников.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateAttestationPriority`

**Параметр:**

* `employee_dict`

**Логика:**

* Получить `previous_result` и `position_changed`.
* Если `position_changed` = `true` → вернуть 3 (высокий приоритет).
* Если `previous_result` = "Неудовлетворительно" → вернуть 2 (средний приоритет).
* Иначе → вернуть 1 (низкий приоритет).

---

### 2️⃣ Функция `DeterminePriorityLabel`

**Параметр:**

* `priority_value`

**Логика:**

* Если `priority_value` = 3 → вернуть `"Высокий приоритет"`
* Если `priority_value` = 2 → вернуть `"Средний приоритет"`
* Иначе → вернуть `"Низкий приоритет"`

---

### 3️⃣ Функция `BuildAttestationRecord`

**Параметры:**

* `employee_name`
* `priority_label`
* `required_status`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "name": "Морозов М.М.",
  "attestation_required": "Требуется",
  "priority": "Высокий приоритет"
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_attestation_list`.
2. В цикле для каждого сотрудника:

   * Вызвать `CheckAttestationDue` (для статуса требования);
   * Вызвать `CalculateAttestationPriority`;
   * Передать результат в `DeterminePriorityLabel`;
   * Вызвать `BuildAttestationRecord`;
   * Добавить возвращённый словарь в `final_attestation_list`.
3. Сохранить `final_attestation_list` в файл
   `attestation_final.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
