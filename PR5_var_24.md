# Вариант 24

## Кейс: Автоматизация проверки корректности товарных накладных

В крупной торговой компании внедряется система автоматической валидации товарных накладных, поступающих от поставщиков.
Руководство приняло решение исключить ручную проверку арифметики документов и минимизировать риски принятия к учёту некорректных накладных.

Ранее сотрудники отдела закупок вручную пересчитывали суммы по строкам, сверяли НДС и итоговые значения, что занимало значительное время при обработке больших партий документов и сопровождалось риском арифметических ошибок.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел документооборота формирует список накладных с данными о позициях и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список накладных;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "invoice_number": "ТН-2024-001",
  "total_amount": 118000,
  "calculated_sum": 100000,
  "vat_rate": 0.18
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка соответствия суммы без НДС

### Ситуация

На первом этапе система должна проверить, корректно ли указана сумма без НДС в накладной.
Для этого необходимо сравнить заявленную `total_amount` с расчётной суммой: `calculated_sum * (1 + vat_rate)`.

Отделу документооборота необходимо получить отдельный файл со статусом проверки арифметики по каждой накладной.

---

## Требуется:

1. Считать JSON-файл (6 накладных).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckInvoiceArithmetic`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `invoice_dict`

**Логика:**

* Получить значения `total_amount`, `calculated_sum`, `vat_rate`.
* Рассчитать ожидаемую сумму:
  `calculated_sum * (1 + vat_rate)`
* Если разница между `total_amount` и расчётной суммой > 1 рубль → вернуть `"Ошибка расчёта"`.
* Иначе → вернуть `"Расчёт верен"`.

---

## Главный процесс должен:

1. Создать пустой список `arithmetic_check_list`.
2. В цикле:

   * Вызвать `CheckInvoiceArithmetic`;
   * Сформировать словарь:

```json
{
  "invoice_number": "ТН-2024-001",
  "arithmetic_status": "Расчёт верен"
}
```

* Добавить словарь в `arithmetic_check_list`.

3. Сохранить `arithmetic_check_list` в файл
   `arithmetic_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку арифметики накладной.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт суммы НДС и формирование итогового статуса

### Ситуация

После проверки арифметики руководство решило автоматизировать выделение суммы НДС и формирование финального статуса документа.

Теперь требуется не только подтвердить корректность расчётов, но и выделить сумму НДС отдельным полем, а также присвоить накладной статус для проведения в учётной системе.

Бухгалтерии нужен структурированный список накладных с расчётными данными для автоматической проводки документов.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 накладных.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateVatAmount`

**Параметр:**

* `invoice_dict`

**Логика:**

* Получить `calculated_sum` и `vat_rate`.
* Рассчитать сумму НДС:
  `calculated_sum * vat_rate`
* Вернуть число (сумма НДС).

---

### 2️⃣ Функция `DetermineInvoiceStatus`

**Параметры:**

* `arithmetic_ok` (булево)
* `vat_amount` (число)

**Логика:**

* Если `arithmetic_ok` = `false` → вернуть `"Отклонена"`.
* Если `vat_amount` = 0 → вернуть `"Без НДС"`.
* Иначе → вернуть `"Принята к учёту"`.

---

### 3️⃣ Функция `BuildInvoiceRecord`

**Параметры:**

* `invoice_number`
* `final_status`
* `vat_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "invoice_number": "ТН-2024-001",
  "processing_status": "Принята к учёту",
  "vat_amount": 18000
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_invoice_list`.
2. В цикле для каждой накладной:

   * Вызвать `CheckInvoiceArithmetic` (получить статус);
   * Вызвать `CalculateVatAmount`;
   * Определить булево значение `arithmetic_ok` на основе статуса;
   * Вызвать `DetermineInvoiceStatus` с двумя параметрами;
   * Вызвать `BuildInvoiceRecord`;
   * Добавить возвращённый словарь в `final_invoice_list`.
3. Сохранить `final_invoice_list` в файл
   `invoice_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
