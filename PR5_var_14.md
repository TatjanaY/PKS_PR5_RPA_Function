# Вариант 14

## Кейс: Автоматизация анализа откликов на рекламную кампанию

В крупной маркетинговой компании внедряется система автоматической обработки результатов рекламных кампаний.
Руководство приняло решение исключить ручную агрегацию данных об откликах и минимизировать риски ошибок при оценке эффективности каналов продвижения.

Ранее маркетологи вручную собирали статистику из разных источников, рассчитывали конверсии и формировали отчёты для клиентов. Процесс занимал значительное время после завершения кампании и сопровождался риском несогласованности данных.

В этом году принято решение автоматизировать процесс с использованием программного робота, разработанного в **Puzzle RPA**.

Отдел аналитики формирует список рекламных каналов с данными об откликах и передаёт его в виде JSON-файла.
Задача разработчика — спроектировать робота с использованием функций, который:

* корректно считывает данные из файла;
* структурированно обрабатывает список каналов;
* разделяет логику на самостоятельные модули;
* формирует новые списки результатов;
* сохраняет итоговые данные в отдельные JSON-файлы.

---

# Требования к исходным данным

1. Исходные данные необходимо **сгенерировать с помощью ИИ DeepSeek**.
2. Сформировать JSON-файл, содержащий **ровно 6 записей**.
3. Формат — список словарей.
4. Структура каждой записи:

```json
{
  "channel_name": "Контекстная реклама",
  "impressions": 50000,
  "clicks": 2500,
  "conversions": 125
}
```

5. Робот должен считать данные из созданного JSON-файла.

---

# 🔹 Задание 1 — Базовый уровень

## Этап 1: Проверка минимального порога кликов

### Ситуация

На первом этапе система должна проверить, достиг ли рекламный канал минимального порога кликов для дальнейшего анализа.
Каналы с количеством кликов менее 1000 считаются недостаточно репрезентативными.

Отделу аналитики необходимо получить отдельный файл со статусом достаточности данных по каждому каналу.

---

## Требуется:

1. Считать JSON-файл (6 каналов).
2. Организовать цикл обработки списка словарей.
3. Создать функцию с возвратом значения.

---

### Функция `CheckClicksThreshold`

**Тип:** Определение функции и возврат значения
**Параметр:**

* `channel_dict`

**Логика:**

* Получить значение `clicks` из словаря.
* Если `clicks` < 1000 → вернуть `"Недостаточно данных"`.
* Если `clicks` ≥ 1000 → вернуть `"Достаточно данных"`.

---

## Главный процесс должен:

1. Создать пустой список `threshold_check_list`.
2. В цикле:

   * Вызвать `CheckClicksThreshold`;
   * Сформировать словарь:

```json
{
  "channel_name": "Контекстная реклама",
  "data_status": "Достаточно данных"
}
```

* Добавить словарь в `threshold_check_list`.

3. Сохранить `threshold_check_list` в файл
   `threshold_check.json`.
4. Вывести итоговый список на экран.

---

## Архитектурные требования

* Функция создаётся через блок «Определение функции и возврат значения».
* Функция выполняет только одну задачу — проверку порога кликов.
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит логики проверки.

---

# 🔹 Задание 2 — Повышенный уровень

## Этап 2: Расчёт конверсии и формирование оценки эффективности

### Ситуация

После проверки достаточности данных руководство решило автоматизировать расчёт ключевых метрик эффективности.

Теперь требуется не только подтвердить репрезентативность, но и рассчитать конверсию, а также присвоить каналу оценку эффективности для принятия решений о дальнейшем бюджете.

Отделу маркетинга нужен структурированный список каналов с расчётными метриками для оптимизации рекламных расходов.

---

## Требуется:

1. Использовать исходный JSON-файл из 6 каналов.
2. Организовать цикл обработки списка.
3. Создать три функции с возвратом значения.

---

### 1️⃣ Функция `CalculateConversionRate`

**Параметр:**

* `channel_dict`

**Логика:**

* Получить `clicks` и `conversions`.
* Если `clicks` = 0 → вернуть 0.
* Рассчитать конверсию:
  `(conversions / clicks) * 100`
* Вернуть число (процент конверсии).

---

### 2️⃣ Функция `DetermineEfficiencyRating`

**Параметр:**

* `conversion_rate`

**Логика:**

* Если `conversion_rate` ≥ 10 → вернуть `"Высокая эффективность"`
* Если `conversion_rate` ≥ 5 → вернуть `"Средняя эффективность"`
* Иначе → вернуть `"Низкая эффективность"`

---

### 3️⃣ Функция `BuildChannelRecord`

**Параметры:**

* `channel_name`
* `efficiency_rating`
* `conversion_value`

**Логика:**

* Создать и вернуть словарь:

```json
{
  "channel_name": "Контекстная реклама",
  "efficiency_rating": "Средняя эффективность",
  "conversion_rate": 5.0
}
```

---

## Главный процесс должен:

1. Создать пустой список `final_channel_list`.
2. В цикле для каждого канала:

   * Вызвать `CalculateConversionRate`;
   * Передать результат в `DetermineEfficiencyRating`;
   * Вызвать `BuildChannelRecord`;
   * Добавить возвращённый словарь в `final_channel_list`.
3. Сохранить `final_channel_list` в файл
   `channel_result.json`.
4. Проверить, что итоговый файл содержит **6 записей**.
5. Вывести итоговый список на экран.

---

## Архитектурные требования

* Все функции создаются через блок «Определение функции и возврат значения».
* Каждая функция реализует принцип «одна функция — одна ответственность».
* Запрещено использовать глобальные переменные.
* Главный процесс не содержит вычислений.
* Используются списки и словари.
* Все результаты обязательно сохраняются в файл.
